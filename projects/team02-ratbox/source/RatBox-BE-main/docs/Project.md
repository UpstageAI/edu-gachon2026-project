# 냉따뚜이 프로젝트 기획서

**참여자**: 김다인, 최서윤

---

## 1. 문제 정의 및 프로젝트 개요

### 프로젝트 한 줄 정의
냉장고 속 재료 정보를 입력받아 AI 에이전트가 스스로 판단하고 도구를 선택해 최적의 레시피와 대체재를 제안하며, 요리가 진행되는 동안에도 사용자의 음성 질의에 실시간으로 대응하는 AI Agent 프로젝트다.

### 서비스 한 줄 정의
냉장고에 있는 재료를 입력하면 그에 맞는 레시피와 부족한 재료의 대체안까지 알려주고, 요리 중에 음성으로 질문하면 그 자리에서 적합한 방법을 텍스트로 알려주는 AI 요리 비서다.

### 서비스 선정 배경
자취생과 1인 가구는 재료를 소량·비정형으로 들쭉날쭉 보유하고 있어, 재료명 완전일치 검색이나 정형화된 필터 위주의 기존 레시피 서비스로는 "지금 가진 재료로 뭘 만들 수 있는지"를 바로 확인하기 어렵다. 매번 재료를 하나하나 검색하거나, 재료가 조금 부족하다는 이유로 레시피 자체를 포기하는 경험이 반복된다. 이 문제는 정형화된 검색이 아니라, 사용자의 상황을 이해하고 대신 판단해주는 AI 에이전트로 풀어야 한다고 봤다.

그리고 이 인터페이스는 텍스트보다 음성일 때 실제 사용 맥락에 더 맞는다. 요리 중에는 손에 물이나 재료가 묻어 있어 텍스트를 입력하기 번거롭고, 사용자는 냉장고 문을 열어 둔 채 눈앞의 재료를 확인하며 즉석에서 말하는 경우가 많기 때문이다. 같은 이유로 PC 기반 웹서비스보다는 항상 휴대하는 모바일 환경이 이 서비스의 실제 사용 맥락에 더 적합하다고 판단해, 모바일 앱과 음성 인터페이스(STT) 기반으로 서비스를 설계했다.

### 해결하려는 문제
**문제**: 사용자가 보유한 재료만으로는 레시피의 재료 목록을 100% 충족하지 못하는 경우가 대부분이며, 이때 "이 재료는 생략해도 되는지", "무엇으로 대체 가능한지"를 스스로 판단하기 어렵다. 이 판단은 요리를 시작하기 전 한 번만 필요한 게 아니라, 조리 중간에 재료가 예상과 다르거나 갑자기 상황이 바뀌었을 때도 반복해서 필요하다.

**문제라고 생각한 이유**: 기존 레시피 서비스는 재료 매칭까지만 제공하고, 그 다음 단계인 부족 재료의 생략·대체 판단은 전적으로 사용자 몫으로 남겨둔다. 게다가 이 판단은 조리를 시작하기 전에 한 번만 끝나는 게 아니라 요리 진행 중에도 계속 발생하는데, 기존 서비스는 조리가 시작된 이후의 상황 변화에는 전혀 대응하지 못한다. 이 판단 과정을 조리 전후 구분 없이 대신해주는 것이야말로 AI가 사용자를 대신할 수 있는 핵심 가치이자, 단순 검색과 AI 에이전트를 구분 짓는 지점이라고 판단했다.

### 대상 사용자
자취생, 1인 가구 등 소량·불규칙한 재료를 보유하고 매 끼니를 스스로 해결해야 하는 사람. 요리 경험이 적어 "이 재료 없어도 되나?"를 판단하기 어려운 초보 자취러를 주요 타깃으로 한다. 특히 조리 중 재료가 부족하거나 예상과 다른 상황이 생기면 검색할 여유 없이 요리를 중단하거나 포기해버리는 경우가 많은 사람을 우선 고려한다.

### 핵심 가치
"재료가 부족해도, 요리 중간에 문제가 생겨도 포기하지 않게 해준다" — 단순히 맞는 레시피를 찾아주는 것을 넘어, 조리를 시작하기 전에는 부족한 재료의 생략·대체 가능 여부를 판단해주고, 조리가 진행되는 동안에도 예상치 못한 상황에 실시간으로 대응해줌으로써, 사용자가 요리를 시작부터 끝까지 완성할 수 있도록 끝까지 함께하는 것.

## 2. 사용자 및 Agent 설계

### 타깃 사용자 페르소나
20대 중후반 자취 3년차 직장인/취준생. 퇴근 후 요리할 시간과 체력은 부족하지만 배달비 부담과 건강 문제로 집밥 비중을 늘리고 싶어 함. 냉장고에는 사다 놓고 다 못 쓴 자투리 재료(대파 반 단, 계란 몇 개, 유통기한 임박 두부 등)가 항상 남아있지만, 이 재료들로 뭘 만들 수 있는지 매번 검색해보는 게 번거로워 결국 배달을 시키거나 재료를 그대로 버리는 경험을 반복함. 막상 요리를 시작해도 레시피 중간에 생각지 못한 재료가 없다는 걸 발견하면, 손에 물이나 재료가 묻어 있어 검색하기도 애매해 어설프게 생략하거나 그대로 조리를 중단해버리는 경우도 잦음.

### Agent의 역할
- 요리비서 Agent인 뚜이는 사용자가 선택한 재료 ID 목록을 받아 레시피 DB에서 매칭 후보를 검색한다.
- 부족 재료 개수가 적은 순으로 레시피 후보 3개를 추천한다.
- 사용자가 그중 하나를 선택하면, 해당 레시피에 한해 부족 재료의 생략 가능 여부와 대체재를 판단해 이유와 함께 설명한다.
- 단순 검색 도구가 아니라 "이 재료는 빼도 되는지, 무엇으로 대체할지"까지 스스로 해석하고 결론 내리는 판단 주체로 기능한다.
- 결과 검증을 통해 최종 제안이 제약을 위반하지 않는지 확인한다. 사용자가 결과에 만족하지 않으면 새 요청으로 다시 검색한다(기존 판단을 이어 수정하는 멀티턴 재판단은 이번 범위에서 다루지 않는다).
- 조리가 시작된 이후에는 사용자의 음성 질의(STT)를 받아, 현재 조리 중인 레시피를 인식한 상태에서 실시간으로 대체재나 생략 가능 여부를 텍스트로 안내한다.

### Agent의 성격 및 톤앤매너
냉장고 구석구석을 뒤져 먹을 걸 찾아내는 부지런하고 귀여운 쥐 캐릭터, 뚜이. 재료가 부족해도 포기하지 않고 "이거랑 이거면 충분해요!"라며 야무지게 대안을 찾아준다. 전문 셰프처럼 딱딱하지 않고, 함께 냉장고를 뒤지는 듯한 캐주얼한 말투를 쓴다 (예: "간장 없어도 괜찮아요, 소금 조금이면 비슷해요"). 요리 초보인 사용자가 위축되지 않도록 판단 근거를 짧고 쉽게 설명하며, 정보 전달자보다는 룸메이트 같은 친근함을 준다.

다만 조리가 진행되는 동안의 응답은 톤은 유지하되 분량은 한두 문장으로 줄인다. 사용자가 손을 쓰던 중에 화면을 잠깐 보고 바로 다시 요리로 돌아가야 하는 상황이므로, 이유 설명보다 "지금 뭘 하면 되는지"를 먼저 준다.

### Agent의 자율성 범위

**스스로 판단하고 실행하는 것**: 레시피 후보 검색, 부족 재료의 생략 가능 여부 판단, 대체재 제안, 추천 순위 결정 및 이유 설명. 조리 진행 중에는 현재 조리 중인 레시피를 인식한 상태에서, 사용자가 음성으로 묻는 재료 대체·생략 여부를 즉시 판단해 답한다.

**스스로 판단하지 않고 사용자에게 맡기는 것**: 실제 장보기·주문 실행, 알레르기·건강 관련 최종 판단(경고 표시까지만 수행), 조리 도구 유무에 따른 레시피 필터링(이번 범위에서 제외). 조리 중 질의에 대해서도 마찬가지로, 대체재가 사용자가 등록한 알레르기 성분에 해당할 가능성이 있으면 자동으로 대체안을 제시하지 않고 경고와 함께 사용자 확인을 요청한다. 또한 조리 중 발화에서 새로운 알레르기 정보가 언급되어도 이를 등록·반영하지 않는다(이번 범위 제외).

**판단 기준 예시**:
- 부족 재료가 1개 이하이고 흔한 조미료(소금, 후추, 식용유 등)에 해당하면 "이미 보유하고 있다"고 가정하고 추천 순위에서 감점하지 않음.
- 핵심 재료(주재료·주단백질)가 2개 이상 없는 경우 해당 레시피는 후순위로 밀리거나 추천에서 제외함.

## 3. 핵심 기능 및 사용자 흐름

### 주요 사용자 시나리오
퇴근 후 지친 자취 3년차 직장인 A씨는 냉장고를 열어보니 계란, 밥, 김치가 남아있다. A씨는 회원가입 시 새우 알레르기를 이미 등록해두었다. 냉따뚜이 앱에서 재료 목록 중 계란, 밥, 김치를 선택하니, 앱이 선택된 재료 ID와 A씨의 알레르기 ID를 함께 전달해 뚜이가 새우가 들어가지 않는 레시피 후보 3개를 부족 재료가 적은 순으로 찾아준다.

A씨는 그중 김치볶음밥을 선택한다. 뚜이는 부족한 재료(간장)의 생략·대체 가능 여부를 판단해 전체 조리 단계와 함께 알려주고, A씨는 조리를 시작한다. 레시피 3단계쯤에서 간장이 없다는 걸 알게 된 A씨가 손에 재료를 든 채로 마이크 버튼을 눌러 "간장이 없는데 어떡하지?"라고 말하면, 뚜이는 현재 조리 중인 레시피를 인식한 상태에서 "소금 반 티스푼으로 대체 가능해요"라는 답을 화면에 텍스트로 띄워준다. A씨는 요리를 멈추지 않고 그대로 이어간다.

### 핵심 기능 정의
- **입력 가드레일**: 요리와 무관하거나 부적절한 입력 필터링
- **재료·알레르기 입력 처리**: 목록에서 선택한 재료 ID와 사용자의 알레르기 ID를 프론트로부터 그대로 전달받아 사용 (자유 텍스트 입력, 동의어 정규화, 서버 측 알레르기 조회 없음)
- **Agent 도구 선택**: LangGraph 기반으로 단계별 도구 호출 — 후보 추천 단계에서는 레시피 검색(Text-to-SQL) 도구만 호출하고, 사용자가 레시피를 선택한 이후에만 부족재료 분류·대체재 검색 도구를 호출한다.
- **결과 검증**: 추천 결과가 사용자 제약을 실제로 위반하지 않는지 재검증
- **출력 가드레일**: 알레르기 유발 재료가 최종 결과에 남아있으면 자동 제외
- **스트리밍 응답**: SSE 기반으로 입력부터 출력까지 실시간 상태 표시
- **조리 중 음성 질의 처리**: 조리 진행 중 STT로 들어온 재료 대체·생략 질문을 인식해, 판단 결과를 즉시 텍스트로 제시 (도구 선택 단계 없이 재료 판단으로 직행)

### 사용자 관점 워크플로우
재료 목록에서 보유 재료 선택 → 실시간 처리 상태 확인 (스트리밍) → 레시피 후보 3개 확인 (요리명, 부족재료 목록만 — 대체재는 아직 미표시) → 후보 중 하나 선택 → 상세보기 진입 (전체 조리법 + 대체재 설명, 조리 모드 시작) → 조리 진행 중 마이크 버튼으로 음성 질의 (예: "간장 없는데 어떡하지?") → 질문받은 재료에 대한 대체안·생략 가능 여부를 짧은 텍스트로 즉시 확인, 요리 계속 → 조리 종료

### 시스템 관점 워크플로우

**A. 텍스트 입력 워크플로우**

*1단계: 후보 추천*
1. [입력] 사용자가 재료 목록에서 선택한 재료 ID 목록과, 프론트가 함께 전달하는 알레르기 ID 목록을 전달받음 → [SSE] 스트리밍 상태 표시 시작
2. [가드레일-입력①] 부적절한 입력 필터링 → 문제가 있으면 즉시 반려, 종료
3. [LangGraph 노드] Text-to-SQL Tool: LLM이 SQL을 직접 생성 → 안전장치(화이트리스트 검증) 통과 후 실행 → 재료 ID 기반 조건부 레시피 후보 검색
4. [필터링] 후보 중 전달받은 알레르기 ID에 해당하는 재료가 포함된 레시피 제외
5. [정렬] 남은 후보를 부족 재료 개수 오름차순으로 정렬 → 상위 3개(레시피명, 부족 재료 목록) 선정
6. [SSE] 후보 3개를 스트리밍으로 출력 (이 시점엔 대체재 판단은 하지 않음)

*2단계: 선택 레시피 상세 판단*
7. [사용자 선택] 프론트에서 3개 중 하나의 `recipe_id`를 선택해 재요청 (알레르기 ID 목록도 함께 재전달)
8. [LangGraph 노드] 분류 Tool: 선택된 레시피의 부족 재료 필수/생략 분류
9. [LangGraph 노드] 대체재 검색 Tool: DB 조회가 아닌 LLM이 레시피 맥락을 반영해 즉석 판단 (생략 불가능한 재료에 한해 수행)
10. [결과 검증 루프] 판단 결과가 전달받은 알레르기 ID를 위반하지 않는지 재검증
11. [가드레일-출력②] 알레르기 재료 필터링 → 최종 조리법·대체재 안내에 알레르기 유발 재료가 있으면 하드 필터링으로 자동 제외
12. [SSE] 레시피명 + 전체 조리 단계 + 대체재 안내를 스트리밍 출력, `recipe_id`를 응답에 포함해 프론트(조리 세션)로 전달

**B. 조리 중 음성 질의 워크플로우**
1. [STT] 조리 모드에서 마이크 입력 → Google Cloud Speech-to-Text로 텍스트 변환
2. [LLM] 발화에서 질문 대상 재료와 의도 추출 (예: "간장 없는데 어떡하지" → 재료: 간장, 의도: 대체재 요청)
3. [정보 전달] 조리 세션이 `recipe_id`와 알레르기 ID 목록을 함께 전달 (서버 측 DB 조회 없음)
4. [LangGraph 노드] 재료 판단 Tool로 직행 (초기 흐름의 도구 선택·분류 단계는 생략 — 이미 어떤 요리인지 정해진 상태이므로 필요 없음)
5. [결과 검증] 전달받은 알레르기 ID와 충돌하는지 확인
6. [텍스트 출력] 검증된 답변을 짧은 텍스트로 즉시 응답 (SSE 스트리밍 없이 단발 응답으로 처리 — 답변이 짧아 스트리밍 이점이 적음)

**워크플로우 구분 목적**
- **입력 가드레일 생략**: 조리 모드에 진입했다는 것 자체가 이미 요리 맥락이 확인된 상태이므로 매번 필터링할 필요가 적음.
- **도구 선택 단계 생략**: A에서는 "어떤 요리를 추천할지" 자체를 몰라서 여러 도구 중 골라야 했지만, B는 이미 요리가 정해진 뒤라 판단 단계를 스킵하고 바로 실행
- **recipe_id 재사용**: 조리 세션(프론트 구현)이 이미 들고 있는 `recipe_id`를 그대로 전달받아 조회하므로, LangGraph가 별도로 대화 상태를 저장할 필요가 없다
- **알레르기 전달 방식**: A와 B 모두 프론트가 알레르기 ID 목록을 요청 파라미터로 함께 전달하는 동일한 방식을 쓴다 — Agent가 별도로 사용자 정보를 조회하지 않아도 되고, 세션 간 상태를 따로 유지(Checkpointer)할 필요도 없는 이유가 된다

## 4. 기술 구현 설계

### 기술 스택
| 영역 | 선택 | 이유 |
|---|---|---|
| LLM | Upstage Solar Pro (Function Calling) | `langchain_upstage.ChatUpstage` 기반 Tool Use, Text-to-SQL 생성 |
| Agent 프레임워크 | LangGraph | 판단→도구선택→재판단 루프를 그래프 노드로 명시적 관리 |
| STT | Google Cloud Speech-to-Text | 조리 중 음성 질의(B흐름) 텍스트 변환 |
| 상태관리 | Stateless 설계 | 별도 그래프 state 저장 없이, 재료/알레르기/레시피 ID를 매 요청 파라미터로 전달받아 필요 시마다 조회 |
| 백엔드 | FastAPI (Python) | 기본 비즈니스 로직 및 SSE 스트리밍 구현 |
| DB | Supabase (PostgreSQL) | 레시피-재료 다대다 관계, Text-to-SQL 쿼리 대상으로 관계형 구조 필요 (필요 시 RAG 및 pgvector 도입 검토) |
| 프론트엔드 | React + SSE | 웹앱 기반 UI, 에이전트의 단계별 판단 과정을 실시간 스트리밍으로 표시 |
| 데이터 소스 | 공공데이터 포털 레시피 DB | 정제 후 PostgreSQL 적재 (1회성 배치처리 사전가공) |
| 관찰가능성 | Langfuse (Cloud SaaS) | Tool 호출 순서·파라미터, LLM 응답, 토큰/지연시간 트레이스 기록 (MVP 이후 연동) |
| 배포/인프라 | GCP Compute Engine (VM) + Docker Compose + Nginx | 백엔드(FastAPI)·프론트엔드(React 빌드본) 단일 VM 내 컨테이너로 구성, Nginx 리버스 프록시로 라우팅 (SSE 사용을 위해 `proxy_buffering off` 설정 필요) |

### 시스템 아키텍처
프론트에서 전달한 재료 ID·알레르기 ID를 기반으로 LangGraph Agent가 도구를 선택해 실행 → 결과 검증 → 자연어 응답. SSE로 실시간 상태를 스트리밍하고, Supabase PostgreSQL에서 레시피 데이터를 검색한다.

구체적으로는 FastAPI 백엔드가 요청을 받아서 LangGraph로 전달하고, LangGraph 내부에서 Text-to-SQL Tool(LLM이 SQL 직접 생성, 안전장치 통과 후 실행), 분류 Tool(필수/생략 판단), 대체재 검색 Tool(LLM 판단) 등이 단계별로 실행된다. 후보 추천 단계에서는 Text-to-SQL Tool만 호출되고, 사용자가 레시피를 선택한 이후에만 분류·대체재 Tool이 호출된다. 각 Tool의 결과는 하나의 요청 안에서 순차적으로 사용되며, 별도 세션 상태로 누적되지 않는다. 최종 결과는 가드레일(알레르기 필터링)을 거친 후 자연어로 변환되어 React 프론트에 SSE로 전송된다.

Day 2-3에는 이 흐름의 최소 버전(후보 검색 → 응답)만 동작시키고, Day 4부터 분류 Tool과 대체재 검색을 추가한다. 아키텍처 자체는 바뀌지 않으며, Tool을 점진적으로 연결하는 방식으로 확장된다.

### 프롬프트 설계 전략
- **역할 분리**: 각 LLM 호출은 하나의 명확한 역할만 한다. "SQL 생성" 프롬프트, "필수/생략 분류" 프롬프트, "응답 생성" 프롬프트를 별개로 만들고, 하나의 프롬프트에 여러 판단을 몰아넣지 않는다.
- **출력 형식 고정**: 판단이 필요한 단계(분류, 검증)는 반드시 구조화된 JSON으로 받는다. 예: `{"필수재료": ["계란"], "생략가능": ["대파"], "이유": "..."}`.
- **근거 명시**: 분류나 판단 결과에는 항상 이유를 함께 출력하게 한다. 이것이 결과 검증 루프와 디버깅을 가능하게 한다.
- **범위 제한**: 시스템 프롬프트에 "요리/재료 관련 질의만 응답. 그 외는 거절"을 명시해서, 입력 가드레일과 이중 방어한다.
- **DB 스키마 컨텍스트 주입**: Text-to-SQL 생성 시 LLM이 참조할 테이블/컬럼 설명을 프롬프트에 함께 전달한다.

### 데이터 활용 및 기억 관리
- **상태 관리 방식**: 알레르기 정보는 프론트가 회원 정보 조회 후 알레르기 ID 목록으로 함께 전달하며, Agent는 별도 DB 조회 없이 전달받은 값을 그대로 사용한다. 세션 간 별도 유지가 불필요하다.
- **장기 저장/개인화 (이번 범위 제외)**: 사용자 계정별 선호 레시피 기록, 과거 알레르기 정보 자동 재사용 등은 이번 개발 범위에서는 다루지 않는다.
- **레시피 데이터**: 공공데이터포털 원본은 1회성 정제 후 DB에 적재하며, 서비스 운영 중 실시간 재수집은 하지 않는다.

### 제약사항 및 예외 처리
- **입력 재료로 매칭 레시피 0건**: "죄송해요, 그 재료 조합으로는 레시피가 없어요. 다른 재료를 추가해주세요." 안내
- **알레르기 재료가 최종 추천에 남음**: 하드 필터링으로 자동 제외 (경고만 하고 남겨두지 않음, 안전 관련 항목이므로 제외가 기본값)
- **알레르기 필터링 후 추천 가능한 레시피가 0건**: "알레르기 재료를 제외하니 추천할 레시피가 없어요. 다른 재료를 추가해주시겠어요?" 안내
- **알레르기 필터링 후 후보가 3개 미만**: 있는 만큼만(1~2개) 제공, 부족하다는 별도 경고는 하지 않음
- **필수 재료를 사용자가 빼달라고 함**: "이 재료가 핵심이라 다른 레시피를 추천합니다" 안내 후 대체 레시피 검색 (Stretch 목표 — MVP 이후 확장 검토)
- **욕설/무관한 입력**: 입력 가드레일에서 즉시 차단, "요리 관련 질의만 도와드려요" 안내
- **LLM 응답이 지정 형식 벗어남**: 파싱 실패 시 즉시 안전 응답(레시피 추천 보류)으로 폴백 (재요청 로직은 MVP 이후 확장 검토)
- **Tool 실행 중 인프라 오류** (DB 연결 실패, LLM API 타임아웃 등): 사용자에게 "일시적인 오류가 발생했어요, 잠시 후 다시 시도해주세요" 안내, 서버 로그에 에러 기록

## 5. 성과 평가 및 실행 계획

### 성공 지표 (KPI)
- **핵심 루프 성공률**: 준비한 테스트 시나리오(재료 조합 10~15개) 중, 재료 입력 → 레시피 추천 → 부족재료 판단(생략/대체)까지 에러 없이 정상 완료되는 비율 — 목표 80% 이상
- **판단 정확도**: 필수/생략가능 재료 분류 및 대체재 제안이 "이 재료 없이 조리 가능한가", "제안된 대체재가 맛/식감상 합리적인가" 2가지 기준에 모두 부합하는지 팀 자체 검수 — 목표 8/10 이상
- **알레르기 필터링 정확도**: 알레르기 유발 재료가 최종 추천 결과에 노출되는 사례 — 목표 0건 (하드 가드레일이므로 예외 허용 안 함)
- **응답 속도 (참고 지표)**: SSE 첫 응답(TTFB) 도달 시간 측정 — 체감 지연 확인용, 별도 목표치 없이 참고자료로만 활용
- **오류개선 효과 (참고 지표)**: Langfuse는 MVP 이후 연동 예정이며, 연동 후 오류분석 미니사이클 적용 전/후 실패 케이스 수를 비교할 계획이다.

### MVP 범위
- 목록에서 선택한 재료 ID·알레르기 ID를 전달
- Text-to-SQL 기반 조건부 레시피 후보 검색 (LLM이 SQL 직접 생성, 안전장치 통과 후 실행)
- 부족 재료 개수 기준 레시피 후보 3개 추천 → 사용자 선택
- 선택된 레시피에 한해 부족 재료 필수/생략 분류 및 대체재 검색·제안
- 판단 근거를 포함한 자연어 응답
- 결과 검증 루프 (알레르기 등 제약 위반 여부 재확인)
- 입력/출력 가드레일 (부적절 입력 차단, 알레르기 재료 하드 필터링)
- SSE 기반 에이전트 판단 과정 실시간 스트리밍 표시
- 조리 중 음성 질의(B흐름): Google Cloud STT 기반 텍스트 변환 + 재료 대체·생략 판단
- Langfuse 트레이스 기반 오류분석 미니사이클 및 Before/After 지표 (계획, MVP 범위 제외)

### 단계별 개발 로드맵

**1단계: 데이터 준비 (Day 1-2)**
- CSV 다운로드 → 정제 → Supabase 적재
- 계획: 50-100개 레시피 / 실제: 8,098건 정제·적재 완료

**2단계: MVP (Day 2-4)**
- 재료 ID 입력 → Text-to-SQL 검색 → 자연어 응답. 기본 한 사이클 동작.
- Day2에 SQL 안전장치·Tool Schema·ReAct Loop 베이스 선착수. Day3-4에 핵심 흐름 테스트 통과까지 완료 목표.
- (2단계 완료 조건: 재료 입력 시 최소 1개 시나리오가 에러 없이 끝까지 동작)

**3단계: 안전장치 & B흐름 (Day 5)**
- 알레르기 하드 필터링 + STT 음성 질의(B흐름) 통합. Checkpointer 없이 stateless로 구현.
- (전제조건: 2단계 완료 조건 충족 후 시작)

**4단계: 검증 (Day 6)**
- 로컬에서 전체 테스트. 테스트 10개로 정확도 확인. 엣지케이스 처리.
- (기준 미달 시: 핵심 루프 성공률 80% 미만이면 배포 전 우선 수정, KPI 목표치와 연동해서 판단)

**5단계: 배포 & 발표 (Day 7)**
- GCP VM 배포 + 발표 준비. 데모 3개 시나리오 실행 확인.
- (배포 이슈 발생 대비, 발표 전날 저녁까지 배포 완료를 데드라인으로 별도 설정 권장)

### 기대 효과
- **사용자 관점**: 재료가 애매하게 부족해서 요리를 포기하던 상황에서, "그래도 만들 수 있다"는 판단과 구체적인 대안을 제공받아 실제 조리로 이어질 가능성을 높인다.
- **기술적 관점**: 단순 검색/추천형 LLM 서비스와 달리, 도구 선택·조건부 분기를 갖춘 Agentic Workflow를 실제로 구현하고 시연함으로써 "LLM 활용 앱"과 "AI Agent"의 차이를 명확히 보여줄 수 있다.
- **팀 관점**: Text-to-SQL, Tool Use, 가드레일, 관찰가능성(Langfuse, 추후 연동), 배포(GCP VM)까지 한 사이클을 짧은 기간 내에 직접 구현해봄으로써 Agent 프로젝트의 전체 구조를 실습 수준으로 체득할 수 있다.
