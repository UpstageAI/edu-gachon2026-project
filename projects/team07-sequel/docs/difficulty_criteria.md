# 난이도 분류 기준 (router)

router 가 질문을 하/중/상/최상으로 나누는 기준. 데이터셋(AI Hub NL2SQL)의
`hardness` 라벨을 **역산**해 정했다 — 별도 가이드 문서는 없고 사실상 Spider
방식(생성 SQL 구성요소 개수) 규칙이다.

## 근거 — 난이도별 평균 SQL 구성요소 (Validation 11,026문항)

| hardness | N | JOIN | WHERE조건 | 집계 | GROUP | ORDER | 서브쿼리 | 컬럼수 |
|---|---|---|---|---|---|---|---|---|
| easy | 2,554 | 0 | 1.0 | 0.05 | 0 | 0 | 0 | 1.0 |
| medium | 5,017 | 0 | 1.2 | 0.08 | 0.05 | 0.08 | 0 | 1.4 |
| hard | 2,729 | 0 | 1.8 | 0.12 | 0.08 | **0.38** | 0 | 1.2 |
| extra hard | 726 | **0.76** | 2.3 | 0.27 | 0.05 | 0.10 | **0.81** | 1.5 |

> 재현: `SP=…/scratchpad; uv run python` 로 VL gold SQL 을 sqlglot 파싱해 집계
> (원 분석 스크립트는 세션 로그 참조). 데이터 갱신 시 재측정.

가장 뚜렷한 경계: **하/중/상은 전부 단일 테이블, 최상만 JOIN·서브쿼리.**
상은 단일 테이블이지만 **정렬(ORDER BY)+다중 조건**(랭킹성)이 특징.

## 기준 정의

| 난이도 | 정의 (생성 SQL 기준) | 질문 표현 신호 |
|---|---|---|
| **하** easy | 단일 테이블, 조건 1개, 집계·정렬 없음 | "X의 Y 알려줘" 단순 조회 |
| **중** medium | 단일 테이블, 여러 컬럼 or 가벼운 집계/그룹 | "몇 개", "합계", "평균", "~별" |
| **상** hard | 단일 테이블, 다중 조건 + 정렬/랭킹 | "가장 많은/적은", "top N", "순위" |
| **최상** extra | 여러 테이블 JOIN or 중첩 비교 | 서로 다른 엔티티 결합, "평균보다", "~중 가장" |

## router 적용 방식

- SQL 생성 **전**이라 질문으로 예측한다. LLM 분류(`prompts.ROUTER_CLASSIFY` 에 위 정의 주입)
  + 앞단 `schema_link` 가 링크한 스키마를 분류기 컨텍스트로 제공해 조인 필요 여부를 판단.
- **주의**: schema_retriever 는 recall 우선이라 항상 여러 테이블을 반환하므로,
  "테이블 수 ≥ 2 → 최상" 같은 **구조 하드룰은 쓰지 않는다**(항상 발동). 조인 필요 판단은 LLM 이 한다.

## 모델 라우팅 (난이도 → 모델) — n=300 검증 확정

우리 실제 generator(few-shot 주입)로 난이도별 300문항 × 3모델 EX 측정
(`bench/route_eval.py`). 결과:

| 난이도 | solar-mini | solar-pro2 | solar-pro3 | → 라우팅 |
|---|---|---|---|---|
| 하 | 81% | **85%** | 81% | **solar-mini** (동급, ~13% 저렴) |
| 중 | 78% | 77% | 78% | **solar-mini** (동률, 저렴) |
| 상 | 73% | **79%** | 72% | **solar-pro2** (정확도 우위) |
| 최상 | 55% | **60%** | 56% | **solar-pro2** (최고, 단 70% 미달) |

결정 근거(비용인지):
- **비용이 입력 지배적** — few-shot 프롬프트(~1500토큰)라 입력비($0.15, 전 모델 동일)가
  지배해 정답당 비용이 3모델 거의 같음 → 정확도가 라우팅을 좌우.
- **하·중은 mini 로 충분**(pro2와 동급), 호출당 ~13% 저렴 → mini.
- **상·최상은 pro2** 정확도 우위(상 +6%p) → pro2.
- **solar-pro3 제외** — 모든 난이도에서 pro2 이하(동일 단가). no-few-shot 실험에서도 동일.

`settings.model_by_difficulty` (env `MODEL_EASY/MEDIUM/HARD/EXTRA_HARD` 로 조정).

> 남은 과제: **최상은 few-shot+pro2 로도 60%**(KPI 70% 미달) — 스키마링킹 강화·예시 수↑·
> 질의 분해 필요. no-few-shot 대비 few-shot 은 전 구간 +12~33%p (few-shot 이 최대 레버).
