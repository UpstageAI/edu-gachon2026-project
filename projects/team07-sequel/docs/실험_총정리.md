## 1. 백본 모델 성적

### 1-1. Solar 3종 비교

> 📊 **AI Hub NL2SQL(한국어)** · 난이도별 300 × 3모델 (구파이프라인, 정적 스키마 + few-shot k3) · mini/pro2/pro3 · 채점: exec_match · 원자료 `results_route*.jsonl`

| 난이도 | solar-mini | solar-pro2 | solar-pro3 | → 라우팅 결정 |
|---|---|---|---|---|
| 하 | 81% | **85%** | 81% | **mini** (동급, ~13% 저렴) |
| 중 | 78% | 77% | 78% | **mini** (동률, 저렴) |
| 상 | 73% | **79%** | 72% | **pro2** (+6%p) |
| 최상 | 55% | **60%** | 56% | **pro2** (최고, 단 KPI 70% 미달) |

- **비용은 입력 지배적**(few-shot 프롬프트 ~1.5k 토큰, 입력단가 3모델 동일 $0.15/1M) → 정답당 비용 거의 같음 → **정확도가 라우팅을 좌우**.
- **solar-pro3 제외** — 모든 난이도에서 pro2 이하(동일 단가). BIRD에서도 pro3 단독 우위는 미확인.

> ⚠️ **이 표의 "→ 라우팅 결정"(하·중 mini)은 구파이프라인 기준의 결론이며, 확정 파이프라인
> 재검증(§7-4)에서 뒤집혔다** — mini 는 신파이프라인의 중 난이도에서 −12pp. 최종 확정은 전 난이도 pro2.

### 1-2. 백본 간 절대 격차

> 📊 **BIRD MiniDev(영어)** · 500 전수 · pro2/pro3 (GPT-4o 는 논문 공표치) · 채점: 공식(set) · 원자료 `results_ablation.jsonl`·`results_bird_full_*.jsonl`

| 백본 | 조건 | EX |
|---|---|---|
| solar-pro2 | 맨몸(zero-shot, 전체 스키마) | **20.4%** |
| **GPT-4o** (논문†) | 맨몸(동일 조건) | **49.8%** |
| solar-pro2 | + metadata | 30.4% |
| GPT-4o (논문†) | + metadata | 61.2% |
| solar-pro3 | 풀 파이프라인(우리 최적) | **51.8%** |
| GPT-4o (논문†) | 풀 파이프라인 + hint | **73.0%** (리더보드 test 77.64) |

## 2. 한국어 AI Hub — 컴포넌트 ablation (정적 벤치)

> 📊 **AI Hub NL2SQL(한국어)** · 난이도별 25 × 3모델 = 조건당 300 · mini/pro2/pro3 · 조건 4개(zero/few/schema/schema+few, 정적 근사) · 채점: exec_match · 원자료 `results*.jsonl`

| 조건 | 전체 EX | 하 | 중 | 상 | 최상 |
|---|---|---|---|---|---|
| zero-shot | **31%** | 48% | 21% | 35% | 19% |
| few-shot | **66%** (+36%p) | 84% | 59% | 59% | 64% |
| schema-linker | **45%** (+14%p) | 53% | 48% | 56% | 23% |
| **schema + few** | **73%** (+42%p) | 77% | 76% | 72% | 67% |

- **few-shot이 최대 단일 레버**(+36%p). 파인튜닝 불가(가격) 판단 후 in-context로 대체 — 유효 확인.

### 2-1. 실 링커 end-to-end 평가 (구파이프라인) — 오염 사건 포함 전말

> 📊 **AI Hub NL2SQL(한국어)** · 1,200 전수 × 3모델 (실 normalize+임베딩 링커) · 채점: exec_match · 원자료 `results_reallinker_base*.jsonl`

정적 근사가 아닌 **실제 normalize→schema_link(임베딩)→generate** 를 AI Hub sqlite 에서 실행
(`bench/route_eval_real.py`).

- **1차 실행은 폐기**: 진단 스크립트를 동시 실행해 상(hard) 전 모델이 정확히 119/300 으로 일치 +
  에러 490건 — 동시 실행이 레이트리밋 오염을 일으킴 (`results_reallinker_base_contaminated.jsonl` 보관).
- **클린 재실행** (3600행, $0.39): 여전히 레이트리밋 321건(중 난이도 집중)이 수치를 깎음.
  에러 제외 EX: **하 60 / 중 59 / 상 61 / 최상 47%** — 정적 근사(§2 표) 대비 **15~20%p 낮음**.
- 원인 추적: normalizer 가 컬럼 개념 구("경찰 1인당 담당 인구" 등)를 keyword 로 뽑고,
  value_retriever 가 이를 임의 셀 값에 억지 매칭("발행 연도"→출판사명 등) → 생성 오도.
  임베딩 자체는 정상 호출됨(진단: 8문항에 78회) — 문제는 매칭 강제였음. → §7-1(컬럼개념 필터)
  · §7-3(확정 힌트만 주입)으로 해결.
- 교훈: **벤치 병렬도·동시 실행 관리 필수** (이후 모든 실험 WORKERS 제한 + 에러행 재개 채움 표준화).

### 2-2. 난이도 분류기(라우터) 실험

> 📊 **AI Hub NL2SQL(한국어)** · 질문 1,200 (정답 라벨 = AI Hub hardness) · 규칙 vs 다수결 분류 정확도

- **규칙 기반 분류 기각**: 질문 텍스트 규칙으로 난이도 분류 시 정확도 **35.4%** — 다수결(전부 "중")
  **45.5%보다 낮음** → 규칙 라우팅 폐기, LLM(solar-mini) 분류 유지.
- **에스컬레이션 라우팅 제안**(미채택): 분류 대신 mini 로 시도→검증/실행 실패 시 pro2 재시도.
  §7-4 에서 전 난이도 pro2 확정으로 **모델 선택 목적의 분류는 무의미**해짐 —
  현재 분류는 로깅·분석용으로만 유지(가드와 1콜 병합, §6).

## 3. BIRD 앵커 — 3-factor ablation (2×2×2 = 8 config)

> 📊 **BIRD MiniDev(영어)** · 500 전수 × 8 config = 4,000 생성 · solar-pro2 고정 (few-shot·evidence off) · 채점: 우리 exec_match + 공식(set) 병기 · $1.64 · 원자료 `results_ablation.jsonl`

**요인**: N=normalize(영어) · L=schema linker(임베딩 top-k) · M=metadata(컬럼 프로파일+LLM 설명)

| config | 우리 채점 | **공식 채점** | 프롬프트 토큰(평균) |
|---|---|---|---|
| N0L0M0 (vanilla) | 19.0% | 20.4% | 721 |
| N0L0M1 | 26.8% | **30.4%** | 4,372 |
| N0L1M0 | 17.4% | 19.4% | 632 |
| N0L1M1 | 23.0% | 26.2% | 3,714 |
| N1L0M0 | 16.8% | 18.4% | 721 |
| **N1L0M1** | **27.6%** | **30.4%** | 4,372 |
| N1L1M0 | 16.8% | 18.2% | 661 |
| N1L1M1 | 21.6% | 24.8% | 3,754 |

**주효과** (우리 채점 기준, on−off 평균):

| factor | Δ | 해석 |
|---|---|---|
| **metadata** | **+7.2pp** | 지배적 레버. 대신 프롬프트 6배(721→4,372 tok) |
| **linker** | **−2.9pp** | BIRD처럼 **스키마가 작으면(3~13테이블) 해로움** — 전체가 컨텍스트에 들어가는데 축소가 recall 손실만 냄. 스키마 크기 의존적 결과 |
| normalize | −0.8pp | 영어·단문이라 중립 (한국어 재측정 필요) |

**채점 방식 차이**: 우리 exec_match(멀티셋·ORDER BY 검사·문자열화)가 공식(set·순서무시·원시튜플)보다 1.4~3.6pp 짜다 → 이후 공식 채점 기본.

## 4. BIRD 성능 개선 — 강화 → 풀 → leave-one-out

> 📊 **BIRD MiniDev(영어)** · 500 전수 (leave-one-out 은 500×6 config) · solar-pro3 + evidence + few-shot8(dev풀, 누수 상한 — §7-2) + metadata · 채점: 공식(set) · 원자료 `results_bird_strong.jsonl`·`results_bird_full_*.jsonl`

### 4-1. 진행 계보 (공식 채점)

| 단계 | 구성 | EX | Δ |
|---|---|---|---|
| ① 베이스 | pro2 + metadata | 30.4% | — |
| ② 강화 | + **pro3 + evidence + few-shot8** | **46.6%** | +16.2 |
| ③ 풀 | + CTE분해 + 다중후보3 투표 + 수리 | 51.2% | +4.6 |
| ④ **최적** | ③에서 **CTE 제외** | **51.8%** | +0.6 |

난이도별(③): simple 69.6 / moderate 46.8 / challenging 35.3. 비용 $1.29/500문항.

### 4-2. Leave-one-out (풀 51.2%에서 하나씩 제거, 각 500문항)

| 제거한 레버 | EX | 기여도 | 판정 |
|---|---|---|---|
| −few-shot | 40.4% | **−10.8pp** | 🥇 최대 (단 dev풀 예시라 상한 추정) |
| −evidence | 43.2% | **−8.0pp** | 🥈 대형 |
| −수리(repair) | 49.4% | −1.8pp | 유효, 소폭 |
| −다중후보 투표 | 50.0% | −1.2pp | 노이즈 경계 · **비용 2.5배**($0.52→$1.29) → 가성비 나쁨 |
| −CTE 분해 가이드 | **51.8%** | **+0.6pp** | ❌ 기각 (빼니 오히려 상승) |

**결론**: 정확도는 **컨텍스트 레버**(few-shot·evidence·metadata)가 지배. 구조 레버(수리·투표·CTE)는 합쳐 ~3pp.

## 6. 토큰 효율 개선 (2026-07-12) — 성능 유지·향상하며 프롬프트 1/3로

> 📊 M-Schema A/B: **BIRD MiniDev(영어)** 150 층화(50×3, seed7) × 2팔 · pro2+metadata · 채점: 공식(set) · 원자료 `results_mschema_ab.jsonl` / 토큰 측정: **Supabase 라이브**

자체 측정: 라이브 스키마 19,014 chars 중 **컬럼 설명이 82%**(평균 135자 문장, 90개 중 82개가
테이블명 재언급), DDL·설명에 컬럼명 전부 중복.

| 적용 | 근거 | 검증 결과 |
|---|---|---|
| **M-Schema 포맷** — DDL+설명 2중구조 → 컬럼당 한 줄 `(col: type, 설명, ex: 샘플)` | XiYan-SQL (arXiv 2411.08599): DDL 대비 4모델 평균 +2.03% | **BIRD 150 A/B: 31.3% → 33.3% (+2.0pp, 논문 수치 재현)** · 에러 0 |
| **설명 간결 재생성** — 135자 문장 → 10자 명사구 + 샘플값 2개 (프로파일 재사용, 재프로파일링 없음) | 위 자체 측정 | **스키마 19,014 → 6,588 chars (65%↓ ≈ 프롬프트 7k→2.5k 토큰)** · E2E 정답 유지 |
| **라우터 2콜→1콜** — 가드+난이도분류를 단일 JSON 판정으로 병합 (fail-closed 유지) | 자체 분석 (질의당 mini 2콜) | 정상 통과·위험 차단 확인 · 질의당 mini 1콜 + ~1초 절감 |
| 프롬프트 캐싱 구조 | Upstage 반복 컨텍스트 60–80% 할인 | 정적 스키마 블록이 이미 프롬프트 앞 = 구조 충족(수정 불요) |

질의당 순효과: **프롬프트 토큰 약 1/3 · LLM 콜 5→4회 · 정확도 동등 이상(+2pp)**.
A/B 원자료: `bench/results_mschema_ab.jsonl` (150문항 × ddl/mschema 2팔, 공식 채점).

> BIRD A/B에서 mschema 팔 토큰이 +9%인 것은 샘플값 추가분(정보량↑) — 실서비스 65% 감소는
> 간결 재생성과 중복 제거가 함께 적용된 실측치.

## 7. 남은 과제 후속 실험 (2026-07-13)

### 7-1. 값 오매칭 노이즈 수정 (완료)

> 📊 **Supabase 라이브(한국어)** · 회귀 라벨셋: 스키마 recall 4 · 값 매칭 4 · 개념 거부 3 (`tests/eval_linker.py`)

실 링커의 고질 노이즈 — normalizer 가 뽑은 **컬럼 개념 키워드**("발행 연도", "결제 수단")가
임의 셀 값에 억지 매칭되던 문제. 임베딩 폴백 단계에 **컬럼개념 필터** 추가:
키워드가 최고 셀 값보다 컬럼명 임베딩에 더 가까우면 값 매칭을 스킵(unresolved).

회귀(`tests/eval_linker.py`): 기존 값 매칭 4/4 유지("신용카드"→credit_card 생존) +
개념 거부 3/3 신설("결제 수단"·"주문 상태"·"계약 유형" → unresolved) — 전부 통과.

### 7-2. few-shot 정석 측정 — dev풀 수치는 크게 낙관적이었다

> 📊 **BIRD MiniDev(영어)** 150 층화(seed7) + **BIRD train 9,428 풀**(HF) · pro3+evidence+meta+few8 · 채점: 공식(set) · 원자료 `results_fewshot_train.jsonl`

BIRD train 풀에서 임베딩 검색으로 정석(누수 없는) few-shot A/B — dev풀(기존)과 동일 150문항·동일 config:

| few-shot 풀 | EX | 해석 |
|---|---|---|
| dev풀 (기존 헤드라인 조건) | 49.3% | 누수 + 같은-DB 예시 → **상한** |
| **train풀 (정석, 누수 없음)** | **36.0%** | 예시가 전부 타 DB → 하한에 가까움 |

격차 −13.3pp 의 원인은 ① 누수 제거 ② **같은-DB 예시의 소멸**(train 풀은 dev DB 를 포함하지 않음).
→ **BIRD 리더보드용 정직한 수치는 36~40% 선**이며, 4장의 46.6~51.8%는 "누수 상한"으로 읽어야 한다.
단, **실서비스는 같은-스키마 예시 풀**(시드 + 쿼리로그)이라 상한 쪽 조건에 가깝다.

### 7-3. 한국어 최종 평가 — normalize 검증과 "불확실 힌트" 발견

### 7-4-1. 라우팅 확정 반영 중 발견한 회귀 버그 (M-Schema × router 절단) — 수정 완료

라우팅 그리드 결과(하·상=mini, 중·최상=pro2, few-shot 난이도별 k)를 `settings.py`에 반영하고
라이브 E2E 로 재검증하는 과정에서 **정상 조회 질문이 안전성 가드에 오차단되는 회귀**를 발견했다.

- 증상: "배송비가 가장 비싼 상위 5개 주문 아이템을 보여줘" 등 명백히 답 가능한 질문이
  `safety.ok=false`("질문에 대한 데이터베이스 조회가 주어지지 않았습니다")로 차단.
- 원인: `router.py` 가 판정용 프롬프트에 스키마를 `schema[:1500]` 로 하드 절단해서 넘기는데,
  **§6 M-Schema 도입으로 스키마가 2,932→6,588자로 커지면서** 11개 테이블 중 앞 2~3개만 보이고
  질문과 관련된 테이블(`olist_order_items.freight_value`, 2296번째 글자)이 잘려나감 →
  LLM 이 "관련 테이블이 안 보인다"를 안전성 문제로 오판해 `ok:false` 반환.
- 즉 **§6 토큰 효율 개선이 이 회귀의 원인** — 스키마를 압축·축소했음에도 컬럼 설명이 늘어난 부분이
  하필 router 의 임의 절단값을 넘겨버린 상호작용 버그.
- 수정: ① `router.py` 의 `schema[:1500]` 절단 제거(전체 전달) ② `ROUTER_JUDGE` 프롬프트에
  "테이블 존재·질문 명확성은 안전성 판단 대상이 아니다" 명시(이중 방어).
- 검증: 오차단 2건 정상 통과(SQL 생성 성공) · easy 질문이 `model=solar-mini` 로 실제 라우팅 확인 ·
  위험 질문("주문 테이블 전부 삭제해줘") 차단은 회귀 없이 유지.
