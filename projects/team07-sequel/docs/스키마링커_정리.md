**AI Hub**(한국어 1,200, 실서비스 조건) 두 개.

## 1. 스키마 링커가 뭔가

파이프라인에서 **normalize 다음, route 앞**에 있는 노드. LLM을 쓰지 않고
(임베딩 + 규칙 코드만) SQL 생성에 필요한 **컨텍스트를 조립**한다.

```
normalize → [스키마 링커] → route → generate → …
                │
                ├─ 테이블 선택   : 질문과 관련된 테이블만 고름 (임베딩)
                ├─ 값 매칭       : 키워드 → 실제 DB 저장값 확정
                ├─ M-Schema 주입 : 컬럼 노트 → 컬럼당 한 줄 렌더 (DDL 대체)
                └─ 예시 검색     : 유사 질문→SQL 예시 최대 8개 확보
```

- **입력**: 정규화된 질문 · 키워드 · 기간(time_range)
- **출력**: `schema` 문자열(아래 ①②③④를 합친 것) + `tables` · `value_hints` · `unresolved` · `fewshot`
- 이 `schema` 문자열을 뒤 단계인 **route(안전성·난이도 판정)와 generate(SQL 생성)가
  프롬프트로 그대로 받는다.** 링커의 품질 = 생성 품질의 상한.

`schema` 문자열의 구성:

```
① M-Schema (컬럼당 한 줄: 타입 + 설명 + 샘플) — 컬럼 노트 없으면 DDL 폴백
② # 조인 경로 (FK)
③ # 값 매칭 (확정)     ← 예: 취소 → orders.status = canceled
④ # 기간               ← time_range (normalize가 규칙으로 계산한 날짜 범위)
```

단, **예시 검색 결과(fewshot)는 이 문자열에 안 들어간다** — state로만 넘기고,
generate가 난이도별 k(하·중 3 / 상·최상 8)로 잘라 프롬프트에 직접 붙인다.
(여기서 검색하는 이유: 재시도 루프에서 중복 검색 방지 + 검색류 작업을 한 노드에 모음)

## 2. 왜 필요한가 — 문제와 부품별 해법

근본 문제: **모델은 DB를 모른다.** 컬럼 이름만 보고 의미를 추측하고, "취소"가 DB에
`canceled`로 저장돼 있는지 `CANCEL`인지 모르고, 테이블이 수백 개면 다 읽지도 못한다.
추측하는 곳마다 SQL이 틀린다. 스키마 링커는 그 **추측을 부품별로 하나씩 제거**한다.

| 부품 | 제거하는 추측 | 방법 |
|---|---|---|
| 테이블 선택 | "어느 테이블을 봐야 하지?" | 질문·테이블 임베딩 양방향 top-k + elbow 컷 + FK로 조인 경로 보강. **단 테이블 ≤ 20개면 선택 자체를 스킵하고 전체 스키마 전달** (Supabase=11개라 해당) |
| 값 매칭 | "'취소'가 DB에 뭐라고 저장돼 있지?" | exact(문자 일치) → synonym(동의어 사전) → fuzzy → embedding 폴백 체인으로 실제 저장값 확인 |
| M-Schema | "이 컬럼이 무슨 의미고 안에 뭐가 들었지?" | 오프라인에서 DB당 1회 추출해둔 컬럼 노트(설명+샘플 2개)를 컬럼당 한 줄로 렌더. 질의당 비용 0 |
| 예시 검색 | "이 DB에선 이런 질문을 어떤 SQL로 쓰지?" | 실행 검증된 유사 예시를 임베딩 검색 |

## 3. 핵심 규칙 3개 (전부 실험으로 확정)

### 규칙 1 — 값 매칭은 **확정 매칭만 프롬프트에 넣는다**

- **확정 매칭** = exact(키워드가 DB 저장값과 문자 그대로 일치) + synonym(사람이 박아둔
  동의어 사전: 취소→canceled). **유사도 점수가 아니라 매칭의 종류**로 가른다 —
  결정론적(동일성·사전)이면 확정, 확률적(유사도)이면 불확실.
- **불확실 매칭**(fuzzy·embedding)은 점수가 아무리 높아도 프롬프트에서 배제,
  `value_hints`(state)로만 보존한다.
- 근거 — **실험 9** (AI Hub 1,200 전수, 주입 방식 비교):

| 주입 방식 | 전체 EX (자체/공식) | Δ vs 안 넣음 |
|---|---|---|
| 값 힌트 안 넣음 (기준) | 72.4 / 75.1% | — |
| ① 불확실까지 전부 "확정(=)"으로 표기해 주입 | 54.6 / 57.5% | **−17.8pp** 💥 |
| ② 불확실은 "후보(단정 금지)"로 표기해 주입 | 63.0 / 65.7% | −9.4pp |
| ③ **확정 매칭만 주입** (채택) | 72.8 / 75.5% | **+0.4pp** ✅ |

  ②가 핵심 교훈: 불확실하다고 **표기해도** 언급 자체가 생성을 오염시킨다.
  → 표기(프레이밍)를 바꾸지 말고 **아예 걸러라**. 이 프로젝트에서 가장 견고한 발견
  (n=1,200, 통계적으로 강하게 유의).
- ⚠️ 소속 주의: 실험 9는 원래 **normalize 검증 실험**이다(기준선 = normalize off).
  "off ↔ ③" 비교(+0.4pp, normalize 켠다)는 **노말라이저 파트** 근거이고,
  이 규칙의 근거는 그중 **①②③ 주입 방식 비교** 슬라이스만이다.

### 규칙 2 — 작은 DB에선 테이블 선택을 **끈다** (소형 우회)

- **실험 3** (BIRD 2×2×2 ablation, 500×8 config): 테이블 선택(T) 주효과 **−2.9pp**.
  스키마가 작으면(3~13테이블) 전체가 컨텍스트에 들어가는데, 선택이 recall만 깎아 해롭다.
- 같은 조건끼리 선택 끔/켬 비교 (BIRD, 자체 / 공식 채점):

| 나머지 조건 | 선택 끔 | 선택 켬 | Δ (공식) |
|---|---|---|---|
| 맨몸 | 19.0 / 20.4% | 17.4 / 19.4% | −1.0pp |
| + 컬럼 노트 | 26.8 / 30.4% | 23.0 / 26.2% | **−4.2pp** |
| + normalize | 16.8 / 18.4% | 16.8 / 18.2% | −0.2pp |
| + normalize + 컬럼 노트 | 27.6 / 30.4% | 21.6 / 24.8% | **−5.6pp** |

  전 조건에서 켜면 같거나 나빠짐(주효과 −2.9pp). 특히 **컬럼 노트가 켜진 조건에서 해가 더 큼** —
  좋은 재료를 선택이 잘라내 버리기 때문.
- → 테이블 ≤ 20개면 선택 스킵, 전체 스키마 전달. Supabase(11개)는 항상 이 경로.
- 같은 실험에서 **컬럼 노트 주효과 +7.2pp** — 링커 재료 중 최대 레버.

### 규칙 3 — 애매하면 SQL을 만들지 말고 **되묻는다**

- **애매 매칭** = embedding 단계에서 top-1과 top-2가 근접(마진 0.05 미만)해
  어느 값인지 확정 못 하는 경우. 후보 top-2를 `candidates`로 보존.
- 그래프가 스키마 링커 직후 분기: 애매 매칭이 있으면 route/generate를 건너뛰고
  후보를 보여주며 되묻는다 (예: '카드' → credit_card · debit_card 중 어느 쪽?).
  잘못된 SQL을 자신 있게 내느니 한 번 묻는 게 낫다 — 규칙 1과 같은 철학.
- normalize의 `ambiguous` 플래그(LLM 판정, 정확도 미측정)는 트리거로 쓰지 않는다.
  "DB에 근접 값이 둘 이상 실재"라는 구체적 근거가 있는 **값 레벨만** 되묻는다.
- 값 매칭 결과별 동작 정리:

| 값 매칭 결과 | 프롬프트 주입 | 이후 동작 |
|---|---|---|
| **확정 매칭** (exact·synonym) | ✅ 주입 | 정상 생성 |
| **불확실 매칭** (fuzzy·embedding) | ❌ 배제 (state 보존) | 정상 생성 — 힌트 없이 |
| **애매 매칭** (top-2 근접) | ❌ 배제 | **되묻기** — 후보 제시, SQL 안 만듦 |
| 매칭 실패 (not_found·unresolved) | — | 정상 생성 — 힌트 없이 |
